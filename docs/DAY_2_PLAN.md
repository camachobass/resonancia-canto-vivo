# Day 2 — Echo becomes an AI musical mentor

## Outcome

By the end of Day 2, a player can submit the song created in the Air world, receive pedagogical feedback generated by GPT-5.6, and immediately hear a playable variation proposed by Echo. The existing curated mentor remains as a transparent fallback if OpenAI is unavailable.

This is the Day 2 vertical slice:

> compose → ask Echo → understand one musical idea → compare two playable versions

We will not open a second world on Day 2. The priority is a thoughtful, visible use of GPT-5.6 that improves the learning loop rather than adding disconnected content.

## Why this is the priority

OpenAI Build Week evaluates technical implementation, design and user experience, potential impact, and idea quality. Strong submissions should demonstrate thoughtful use of GPT-5.6 and Codex. The current code already has the ideal seam: `POST /api/mentor/feedback` is validated with Zod but still returns curated mock content.

Day 2 replaces that internal implementation without breaking the Day 1 game.

## Player experience

1. The player composes the dog walk as today.
2. The portal shows a short “Echo is listening…” state.
3. Echo explains how tempo, contour, and instrumentation affect the character of the song.
4. Echo proposes one achievable variation using only controls the game already supports.
5. The player can switch between:
   - **My song**
   - **Echo’s variation**
6. Both versions are rendered by Tone.js, so the AI suggestion becomes audible rather than remaining text.
7. The interface identifies whether feedback came from GPT-5.6 or from the curated fallback.

## Technical contract

### Request

```ts
{
  sceneId: "dog-walk";
  language: "en" | "es";
  playerSessionId: string; // random local UUID, never a name or email
  composition: {
    tempo: 72 | 96 | 120;
    contour: "rising" | "falling" | "wave";
    instruments: ("pluck" | "flute" | "bass" | "percussion")[];
  };
}
```

### Response

```ts
{
  interpretation: string;
  concept: string;
  variationChallenge: string;
  worldEffect: string;
  suggestedVariation: {
    tempo: 72 | 96 | 120;
    contour: "rising" | "falling" | "wave";
    instruments: ("pluck" | "flute" | "bass" | "percussion")[];
    reason: string;
  };
  source: "openai" | "mock";
  model?: string;
}
```

The server will use the official OpenAI JavaScript SDK, the Responses API, and Structured Outputs derived from the same Zod schema used by the application. No API key or OpenAI request will run in the browser.

## Model strategy

- Default model: `gpt-5.6` through a server-only environment variable.
- Reasoning: low, because the task is constrained and latency matters.
- Short structured output to control latency and cost.
- Model name configurable through `OPENAI_MENTOR_MODEL` without changing code.
- No free-form player text on Day 2; GPT receives only constrained musical parameters and language.

## Execution schedule

### Block 1 — Secure configuration · 30 minutes

- Create the OpenAI project key through the secure Codex/OpenAI Platform flow.
- Add `OPENAI_API_KEY` and `OPENAI_MENTOR_MODEL` to Vercel.
- Add `.env.example` with names only, never secrets.
- Confirm secrets are excluded from Git.

**Gate:** a server-only smoke request succeeds locally or in a protected preview.

### Block 2 — Live mentor endpoint · 90 minutes

- Install the official `openai` package.
- Extend the Zod response schema with `suggestedVariation`.
- Build a concise bilingual music-pedagogy prompt.
- Call `openai.responses.parse` with Structured Outputs.
- Detect refusals, incomplete output, timeouts, and SDK errors.
- Preserve the curated mock as fallback with `source: "mock"`.

**Gate:** valid requests always return the documented response shape.

### Block 3 — Audible AI variation · 90 minutes

- Add the listening/loading state to the portal transition.
- Display a clear “Powered by GPT-5.6” or “Curated fallback” badge.
- Add “My song” and “Echo’s variation” controls.
- Send `suggestedVariation` through the existing Tone.js composition engine.
- Visually distinguish the selected version and update the path animation.

**Gate:** a player can hear both versions without reloading or losing progress.

### Block 4 — Reliability and child-safe boundaries · 45 minutes

- Generate a random anonymous session UUID in local storage.
- Derive a privacy-preserving safety identifier on the server.
- Limit payload size and output tokens.
- Add a request timeout and a friendly fallback state.
- Do not collect names, emails, microphone recordings, or free-form child input.

**Gate:** removing the API key or simulating an API error still completes the game through the disclosed fallback.

### Block 5 — Tests and pedagogy evals · 75 minutes

- Unit tests for the extended request and response schemas.
- API tests for OpenAI success, refusal, malformed output, timeout, and fallback.
- Audio mapping test proving the suggested variation is playable.
- Bilingual copy parity test.
- Create a small evaluation set covering both languages, all three contours, tempo extremes, and one/three-instrument arrangements.

Every mentor response must:

1. Name at least one choice the player actually made.
2. Explain one age-appropriate musical relationship.
3. Propose a variation available in the current controls.
4. Avoid pretending to have heard microphone audio.
5. Stay concise enough to read without interrupting play.

**Gate:** schema success is 100% across the evaluation set and all existing Day 1 tests still pass.

### Block 6 — Production verification · 60 minutes

- Run lint, unit tests, production build, and the evaluation set.
- Test the full English and Spanish paths in the browser.
- Verify desktop and 390×844 mobile layouts.
- Deploy to the existing Vercel project.
- Verify `/`, `/api/health`, a real OpenAI response, and the fallback path.
- Update README with the Day 2 architecture and an honest disclosure of the model/fallback behavior.

**Gate:** the public URL completes the entire Day 1 flow plus the GPT-5.6 comparison loop.

## Definition of done

Day 2 is complete only when all of these are true:

- GPT-5.6 produces the mentor response in production.
- The response is schema-validated and visibly labeled as OpenAI-generated.
- Echo’s proposed variation can be played with Tone.js.
- Original and variation can be compared in the same scene.
- Mock fallback is automatic, tested, and honestly labeled.
- No secret or personally identifiable information reaches the repository or browser bundle.
- English and Spanish paths work.
- Lint, tests, build, browser verification, GitHub push, and Vercel deployment pass.
